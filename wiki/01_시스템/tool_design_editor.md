# GuildMaster Data Editor — 기획서

> 본 문서는 GuildMaster 프로젝트의 JSON 기반 데이터 드리븐 아키텍처를 위한 외장 에디터 도구의 기획/의견서입니다.
> **현재 단계**: 기획/설계 (제작 미착수)

---

## 1. 개요 및 목적

### 1.1 문제 정의

현재 프로젝트의 모든 게임 콘텐츠(장소, NPC, 대화, 상호작용, 이벤트)는 JSON 파일로 관리됩니다. `planner-guide.md`에 기술된 대로, 기획자는 JSON을 손으로 직접 작성합니다.

**현재의 불편함:**
- JSON 문법 오류(쉼표, 따옴표, 중괄호)에 취약
- 파일 간 참조 무결성 확인 어려움 (존재하지 않는 `place_id`를 적었는지 쉽게 놓침)
- 대량의 항목 관리 시 가독성 저하
- 텍스트 필드 내 `\n`을 포함한 멀티라인 문자열 편집이 불편
- 항목 순서 변경 시 파일명/내용 일일이 수정 필요

### 1.2 도입 목적

**비프로그래머 기획자가 손쉽게 데이터를 생성·편집·관리**할 수 있는 시각적 에디터를 제공. JSON의 기계적 문법을 숨기고, **데이터의 의미(semantic)**만 다루게 한다.

## 2. 타겟 사용자 및 사용 시나리오

### 2.1 타겟 사용자

| 사용자 | 역할 | 핵심 니즈 |
|--------|------|----------|
| **기획자/시나리오 라이터** | 대사, 이벤트, NPC 스케줄 작성 | 대량 대사 입력, 분기 조건 설정, 멀티라인 텍스트 편집 |
| **메인 개발자 (본인)** | 시스템 구조 확장, JSON 스키마 정의 | 스키마 변경 시 에디터 대응, 데이터 무결성 검증 |

### 2.2 핵심 사용 시나리오

**시나리오 A — "루이제의 새로운 아침 대사 추가"**
```
1. 에디터 실행 → 좌측 패널에서 data/dialogues/ 폴더 열기
2. luise_morning.dialogue 클릭 → 우측 편집 영역에서 대사 텍스트 수정
3. 멀티라인 텍스트 필드에 자연스럽게 줄바꿈 입력 (에디터가 내부적으로 \n 처리)
4. 저장 버튼 클릭 → JSON 파일이 자동으로 갱신됨
```

**시나리오 B — "새로운 NPC 셰퍼드 추가"**
```
1. 좌측 패널 우클릭 → "새 NPC 스케줄" 선택
2. 이름, 표시 이름, 초상화 경로 입력
3. 스케줄 행을 추가하여 장소/시간대/가중치 설정
   - 장소 선택 시 드롭다운으로 유효한 place_id 목록 표시
   - 시간대 선택 시 체크박스로 morning/afternoon/evening/night 선택
4. 저장 시 data/npc_schedules/shepard.json 자동 생성
```

**시나리오 C — "이벤트 연결 확인"**
```
1. 좌측 상단 검색창에 "lobby" 입력
2. 모든 place_id가 "lobby"인 장소, 이벤트, NPC 스케줄이 필터링되어 표시
3. 더블클릭으로 해당 파일 열어 편집
```

```
┌─────────────────────────────────────────────────────────────┐
│  [프로젝트 경로]  [저장] [새로고침] [검증] [설정]          │  ← 툴바
├──────────────────┬────────────────────────────────────────┤
│                  │                                        │
│  [🔍 검색창]      │                                        │
│  ───────────────  │     ┌─────────────────────────────┐  │

│  │  ├─ lobby       │     │                              │  │
```

### 3.2 좌측 패널 — 탐색/검색

**폴더 트리:**
- 프로젝트의 `data/` 폴더를 미러링한 트리 구조
- 폴더는 접기/펼치기(▶/▼) 가능
- 파일은 아이콘으로 타입 구분 (장소 🏠, NPC 👤, 대화 💬, 이벤트 ⚡, 아이템 📦)
- 드래그 앤 드롭으로 파일 이동/순서 변경 가능
- 우클릭 컨텍스트 메뉴: 새 파일, 복제, 삭제, 이름 변경

**검색창:**
- 실시간 필터링 (입력할 때마다 트리 갱신)
- 검색 범위: 파일명, 내부 텍스트, ID, 태그
- 고급 검색: `place_id:lobby type:event` 같은 필터 구문 지원

### 3.3 우측 패널 — 편집 영역

**듀얼 탭:**
- **"설계 보기" 탭 (기본)**: 폼 기반 UI. 필드별 입력 컨트롤 제공
- **"JSON 원본" 탭**: 포맷팅된 JSON을 직접 편집 가능. 전문가용

**폼 기반 편집의 핵심 규칙:**
- `id`/`place_id`/`npc_id` 등 ID 필드: 텍스트 입력 + 유효성 검사 (파일명과 동기화)
- 참조 필드 (`connections`, `place_id` in schedules): 드롭다운 + 다중 선택. 존재하지 않는 ID 입력 시 경고
- 텍스트 필드 (`display_name`, `message`): 단일행 텍스트
- 멀티라인 텍스트 (`description`, 대사 본문): 텍스트 에리어. **줄바꿈을 자연스럽게 입력 가능** (내부적으로 `\n` 변환)
- 숫자 필드 (`weight`, `priority`, `amount`): 스핀박스
- 배열 필드 (`connections`, `actions`, `events`): 동적 추가/제거 가능한 리스트 UI
- 조건 필드 (`when`, `conditions`): 조건 빌더 UI (드롭다운 + 값 입력 + AND/OR 조합)

## 4. 데이터 모델 및 프로젝트 연동

### 4.1 지원 파일 타입

| 폴더 | 확장자 | 스키마 | 에디터 타입 |
|------|--------|--------|------------|
| `data/places/` | `.json` | Place | PlaceEditor |
| `data/npc_schedules/` | `.json` | NPC Schedule | NPCScheduleEditor |
| `data/items/` | `.json` | Item | ItemEditor |
| `data/interactions/` | `.json` | Interaction | InteractionEditor |
| `data/events/` | `.json` | Event | EventEditor |
| `data/dialogues/` | `.dialogue` | DialogueManager 문법 | DialogueEditor |
| `data/actions/` | `.json` | 재사용 Action 매크로 | ActionEditor |

### 4.2 스키마 기반 폼 생성

에디터는 하드코딩된 폼이 아니라, **JSON 스키마**를 읽어서 동적으로 폼을 생성한다.

```json
// schemas/place.schema.json (에디터 전용)
{
  "type": "object",
  "fields": [
    { "key": "place_id", "type": "id", "label": "장소 ID", "required": true },
    { "key": "display_name", "type": "text", "label": "표시 이름", "required": true },
    { "key": "background_path", "type": "path", "label": "배경 그림", "base_dir": "res://assets/bg/" },
    { "key": "bgm", "type": "select", "label": "BGM", "options_from": "bgm_registry" },
    { "key": "empty_weight", "type": "number", "label": "빈 장소 가중치", "min": 0, "default": 10 },
    { "key": "connections", "type": "id_list", "label": "이동 가능 장소", "ref_type": "place" },
    { "key": "tags", "type": "tag_list", "label": "태그" }
  ]
}
```

스키마 파일은 에디터와 별도로 관리되며, 프로젝트의 데이터 구조가 확장되면 스키마만 갱신하면 에디터 UI가 자동으로 대응한다.

### 4.3 참조 무결성 검증 (Link Validation)

에디터는 다음 참조를 실시간으로 검증한다:

| 검증 항목 | 예시 |
|-----------|------|
| 장소 참조 | `connections`의 `place_id`가 `data/places/`에 존재하는가 |
| NPC 참조 | `npc_id`가 `data/npc_schedules/`에 존재하는가 |
| 대화 참조 | `dialogue_id`가 `data/dialogues/`에 존재하는가 |
| 아이템 참조 | `item_id`가 `data/items/`에 존재하는가 |
| BGM 참조 | `bgm` 값이 등록된 BGM 목록에 존재하는가 |
| 그림 경로 | `background_path`, `portrait_path`가 실제 파일로 존재하는가 |

**검증 결과 표시:**
- 유효한 참조: ✅ 또는 아무 표시 없음
- 누락된 참조: ⚠️ 노란색 경고 (빨간색으로 강조)
- 저장 시 누락 참조가 있으면 경고 다이얼로그 + 저장은 가능 (개발 중 유연성)

## 6. 기술 스택

| 구성 요소 | 기술 | 이유 |
|-----------|------|------|
| **프레임워크** | Electron + React (또는 Vue/Svelte) | 크로스플랫폼, 풍부한 UI 컴포넌트 생태계 |
| **JSON 처리** | JSON5 (또는 표준 JSON) | 주석 지원, 더 유연한 파싱 |
| **스키마 검증** | JSON Schema + Ajv | 동적 폼 생성, 유효성 검사 |
| **파일 시스템** | Node.js fs + chokidar | 실시간 파일 감시, 자동 갱신 |
| **diff/undo** | json-diff-patch | JSON 구조의 세밀한 변경 추적 |
| **빌드** | electron-builder | Windows/Mac 배포 |

**대안 고려:**
- Tauri (Rust 기반, 가벼움) — Electron 대비 용량이 훨씬 작지만 Rust 의존성
- VS Code Extension — 이미 VS Code를 사용한다면 확장으로 개발 가능. 하지만 비개발자 기획자에게는 진입장벽

## 8. 개발 로드맵 (참고)

**Phase 1 — MVP (2주)**
- Electron + React 기본 구조
- 폴더 탐색 트리 + JSON 편집 폼
- Place, NPC, Interaction 3가지 타입 지원
- 기본 저장/검증

**Phase 2 — 확장 (2주)**
- 동적 스키마 로딩
- Dialogue .dialogue 파일 지원
- 드래그 앤 드롭, 고급 검색
- Event, Item 타입 추가

**Phase 3 — 고급 (2주)**
- 미리보기 기능
- 변경 내역 추적
- 프로젝트 설정 파일 (에디터 전용 설정)
- 패키징/배포

## 10. 의견 및 제언

### 10.1 즉시 제작해야 하는가?

**지금 당장은 아니다.** 현재 기획자(본인)가 직접 JSON을 작성하고 있으며, 항목 수가 아직 적음(장소 9개, NPC 2개). 에디터의 ROI는 데이터량이 50개 이상 넘어갈 때 발생한다.

**하지만 2-4주 후에는 고려해야 한다.** 루이제 콘텐츠 + 추가 NPC 2명 + 던전 이벤트 + 엔딩 분기가 추가되면 JSON 파일 수가 100개 이상으로 늘어난다. 그때부터 손작성의 비효율이 누적된다.

### 10.2 대안 도구 검토

VS Code + JSON Schema Extension을 활용하는 방법도 있다. JSON Schema를 작성하면 VS Code에서 자동 완성, 유효성 검사가 가능하다. 하지만:
- 멀티라인 텍스트의 자연스러운 편집은 불가
- 드래그 앤 드롭 순서 변경은 불가
- 폼 기반 UI가 아님

따라서 **전용 에디터의 가치는 충분하다.** 단, VS Code Extension으로 먼저 시험해보는 것도 Phase 0으로서 유효하다.

### 10.3 핵심 가치 재확인

이 에디터의 성공 기준은 단 하나:

> **"기획자가 JSON 문법을 생각하지 않고, 오직 게임의 의미(누가, 어디서, 무엇을, 어떤 조건으로)만 다룰 수 있는가?"**

이 기준을 만족하는 MVP부터 시작하여, 점진적으로 기능을 쌓아가는 것이 바람직하다.
